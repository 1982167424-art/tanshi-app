## 线上部署修复（关键！代码改了但线上没生效）
- 现象：改完 api.ts 直连 + token 修复后，用户反馈"API还是请求失败"，线上 JS 还是旧版 index-vV3CGXFI.js（不含 api.textime.top）
- 根因有三个层面：
  1. **git 未 commit/push**：Vercel CLI 用 `git ls-files` 上传文件，未 commit 的修改不会被上传。之前所有 `vercel --prod` 部署用的都是 git HEAD 旧代码
  2. **git push 后 Vercel 未自动部署**：项目 GitHub 仓库 (github.com/1982167424-art/tanshi-app) 未配置 Vercel 自动部署 webhook
  3. **Vercel build cache**：即使 commit 后 `vercel --prod` 重新部署，Vercel 恢复了 build cache 导致构建出旧 hash（index-vV3CGXFI.js 而非 index-Bg5HpVuJ.js）
- 最终解决方案：`vercel pull --yes` → `vercel build --prod`（本地构建生成 .vercel/output）→ `vercel deploy --prod --prebuilt --yes`（直接上传本地构建产物，跳过 Vercel 构建）
- 同时修改 vercel.json：HTML 的 Cache-Control 从 `public, max-age=300, s-maxage=300` 改为 `public, max-age=0, s-maxage=0, must-revalidate`，避免部署后 HTML 被 CDN 缓存 5 分钟
- 验证通过：线上 JS 更新为 index-Bg5HpVuJ.js，api.textime.top 出现 1 次；跨域 GET /api/health → 200 JSON；OPTIONS 预检 → 204；CORS allow-origin: https://textime.top
- **重要教训**：改完代码后必须 git commit + push，且 Vercel 部署用 prebuilt 模式最可靠（避免 build cache 问题）

## 修复 AI 聊天请求超时
- 现象：用户反馈"AI API调用失败: Error: 请求超时"
- 根因：前端 fetchWithTimeout 默认 15 秒超时，但后端 AI 服务调用 pollinations.ai（免费 AI API）可能需要更长时间，加上后端有 30 秒超时 + 2 次重试（最多约 93 秒），前端 15 秒就先超时了
- pollinations.ai API 实测可用，正常响应约 3 秒，但偶尔可能慢
- 修复：api.ts 的 request 函数增加 timeoutMs 参数（默认 15 秒），AI chat 和 analysis 请求设为 60 秒超时
- 验证：tsc -b 通过；线上 JS 含 6e4（60000 压缩形式）2 次；commit d991b15，prebuilt 部署成功

## 修复注销账号被限流误拦
- 现象：用户报注销账号时"操作过于频繁"
- 根因：server.js 的 sensitiveLimiter 对 `/api/users` 所有方法限流（每IP每小时3次），但 GET 同步用户信息是登录后自动触发的读取操作，不该占用敏感配额，导致 PUT/DELETE 真正敏感操作被挤掉
- 修复：sensitiveLimiter 的 skip 增加 `|| req.method === 'GET'`，GET 不计入限流，PUT/DELETE 保持原样
- 验证：Railway 部署 SUCCESS (11:00:35)；/api/health 200 JSON；Express 正常启动
- commit 7808da2

## Turnstile 验证组件国内无法加载的兜底方案
